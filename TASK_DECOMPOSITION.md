## 强制拆解规则

### 触发条件

当任务满足以下任一条件时，**必须**拆解：

| 条件 | 示例 |
|------|------|
| 涉及 ≥3 个文件 | "实现文件上传功能" |
| 涉及 ≥2 个组件 | "Agent + Server 都要改" |
| 涉及协议变更 | "新增消息类型" |
| 包含多个功能点 | "实现 PTY + 文件传输" |
| 预估代码 ≥200 行 | 大规模重构 |

### 拆解步骤

```
1. 识别子任务
   → 列出所有子任务，每个 ≤50 行代码
   
2. 确定依赖关系
   → 哪些任务必须先完成
   
3. 按顺序执行
   → 每完成一个，再开始下一个
   
4. 每个子任务独立测试
   → 不等所有完成再测试
```

当收到大任务时，先输出：

```markdown

### 子任务 2: [名称]

- **依赖：** 子任务 1
- **测试：** [测试方法]

**执行顺序：** 1 → 2 → 3 → ...

**是否开始执行？**（等待确认）
```

## 上下文管理策略

### 1. 分阶段输出

```
阶段 1: 设计（只输出设计，不写代码）
  → 用户确认后再进入下一阶段

阶段 2: 实现（按子任务逐个实现）
  → 每个子任务完成后，等待用户确认

阶段 3: 测试（独立测试阶段）
  → 运行测试，报告结果
```

### 2. 精简上下文

```markdown
## 输出要求

- 代码只输出改动部分，不输出整个文件
- 使用 diff 格式标注改动位置
- 删除的代码用注释标注：// REMOVED: ...
```

### 3. 子任务委托

对于独立的子任务，可以委托给 sub-agent：

```
"子任务 3 (Agent 状态机重构) 委托给子会话处理"
→ 创建新会话，只传递：
    - 具体任务描述
    - 相关文件路径
    - 接口约定
→ 完成后合并结果
```

## 大任务处理流程

```
用户发起大任务
    ↓
检查是否需要拆解
    ├─ 是 → 输出拆解计划
    │        ↓
    │      等待用户确认
    │        ↓
    │      按顺序执行子任务
    │        ↓
    │      每个子任务完成后确认
    │
    └─ 否 → 直接执行
```

## 防止质量下降的技巧

### 1. 单一职责

```
每个子任务只做一件事：
✓ "添加 FILE_UPLOAD 消息类型"
✗ "添加消息类型并实现 Agent 逻辑并写测试"
```

### 2. 明确边界

```
明确输入输出：
- 输入：消息类型定义、数据结构
- 输出：函数实现、测试通过
```

### 3. 独立测试

```
每个子任务都有独立的测试：
- 不依赖其他子任务的测试
- 可以单独验证
```

### 4. 最小依赖

```
尽量减少子任务之间的依赖：
- 可以并行的尽量并行
- 必须串行的明确标注
```

## 常见大任务类型及拆解建议

| 任务类型 | 拆解方式 |
|----------|----------|
| 新增功能 | 按组件拆：协议 → Agent → Server → Web |
| 重构 | 按模块拆：接口定义 → 实现 → 测试 |
| Bug 修复 | 按步骤拆：定位 → 分析 → 修复 → 验证 |
| 性能优化 | 按瓶颈拆：分析 → 优化点1 → 优化点2 → 验证 |
| 文档更新 | 按文件拆：PROTOCOL.md → README → 注释 |

## 自动拆解机制（给 AI 助手）

### 强制触发条件

当检测到以下信号时，**立即**停止并输出拆解计划：

```
信号 1: 需要修改的文件数 ≥ 3
信号 2: 任务描述包含"和"、"并且"、"同时"等连接词
信号 3: 涉及多个组件（Agent + Server + Web）
信号 4: 预估代码量 ≥ 200 行
信号 5: 需要新增消息类型（涉及协议同步）
```

### 自动拆解流程

```
步骤 1: 解析任务
  → 识别所有涉及的功能点
  → 列出需要修改的文件
  
步骤 2: 分组
  → 按组件分组（Agent / Server / Web / Protocol）
  → 按依赖关系排序
  
步骤 3: 输出计划
  → 使用标准模板
  → 等待用户确认
  
步骤 4: 执行
  → 逐个子任务执行
  → 每个完成后确认
```

### 子任务委托

对于大型子任务，使用 sub-agent 机制：

```markdown
## 子任务委托模板

当前任务过于复杂，委托子任务 [X] 给子会话处理。

**委托内容：**
- 任务：[具体描述]
- 文件：[文件路径]
- 接口：[输入输出约定]
- 测试：[测试要求]

**创建子会话...**
```

## 上下文精简技巧

### 1. 差分输出

```markdown
## 输出格式

只输出改动的代码片段，格式：

```diff
// 文件：src/agent.c
// 位置：handle_message() 函数

@@
     case MSG_TYPE_HEARTBEAT:
         return handle_heartbeat(ctx, data);
+    case MSG_TYPE_FILE_UPLOAD:
+        return handle_file_upload(ctx, data);
     default:
         LOG_WARN("未知消息类型: 0x%02X", msg_type);
@@
```

### 2. 引用而非重复

```markdown
## 引用现有代码

见 PROTOCOL.md 第 4.3 节 FILE_REQUEST 定义
见 STYLE.md "C 代码规范" 章节
见 agent.h 第 45 行 msg_type_t 定义
```

### 3. 分阶段确认

```
阶段 1: 设计输出
  → 等待用户确认
  
阶段 2: 实现输出
  → 用户确认后输出代码
  
阶段 3: 测试输出
  → 用户确认后运行测试
```

## 反模式（避免）

```
❌ 一个响应包含所有代码
❌ 重复输出完整的文件内容
❌ 多个功能混在一起实现
❌ 不等待确认直接执行
❌ 上下文超过 10000 tokens 仍继续
```
